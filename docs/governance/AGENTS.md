# AGENTS.md — AI Agent 协作标准与工作流（内部执行规范）

> **版本：** 1.3
> **地位：** Agent 行为的**物理护栏**。

---

## 1. 优先级与决策逻辑

### 1.1 双重裁决模型
1. **治理规则顺序：** `constitution.md` > `AGENTS.md`
2. **业务真相顺序：** `spec.md` > `plan.md` > `tasks.md` > 既有代码

### 1.2 裁决原则
Agent 必须同时满足两类约束：
- **治理约束：** 是否符合宪法与SOP规定的协作、测试、同步与变更纪律。
- **事实约束：** 是否符合 `spec.md` 定义的业务行为，以及 `plan.md` 定义的技术方案。

若两类约束出现冲突，Agent 不得在代码层自行拍板，必须停止实现并回到文档层澄清或修正。若 `spec.md` 存在歧义或 `plan.md` 过度设计，Agent 必须在编码前提出异议。

---

## 2. 编码前的思考 (Think Before Coding)

**不要假设，不要隐藏困惑，显性化权衡。**

在实现任何任务前，Agent 必须：
- **声明假设：** 显式列出你的理解前提。若有不确定，立即提问。
- **呈现选择：** 存在多种解释或实现路径时，列出候选方案及其优劣，严禁静默选定。
- **推回复杂度：** 若发现更简单的路径，必须提出。问自己："资深工程师会觉得这太复杂了吗？"如果是，请重写。
- **升级阈值：** 只有当发现会改变业务规则、外部行为、关键契约或核心技术约束时，才升级到 `spec.md` / `plan.md`；纯实现细节、命名、局部结构与测试组织问题，不应触发文档回写。
- **熔断：** 只要有不清晰的地方，立即停止。指出模糊点，等待澄清。

---

## 3. 目标驱动执行 (Goal-Driven Loop)

**定义成功标准，循环直至验证。**

将任务转化为可验证的目标（Verifiable Goals）：
- **新增功能 / Bug 修复：** 先写失败测试，再实现，直至通过。
- **行为不变的重构 / 工程化调整：** 先定义不变性与验证方式，再实施，确保回归通过。
- **文档 / 配置任务：** 先定义一致性检查点，再修改，验证结果与目标一致。

**多步骤任务执行格式：**
每次开始复杂任务前，必须输出简要计划：
> 1. [步骤 A] → 验证方式：[具体检查点]
> 2. [步骤 B] → 验证方式：[具体检查点]

---

## 4. 极简主义 (Simplicity First)

**用解决问题所需的最小代码量，不做任何投机性设计。**

- **严禁预判：** 禁止实现未被明确要求的任何功能。
- **拒绝抽象：** 禁止为单次使用的代码创建抽象层；禁止增加未要求的"灵活性"或"配置项"。
- **缩减率：** 若能用 50 行代码解决问题，严禁写 200 行。
- **MECE 审计：** 在处理分类时，必须执行"互斥且穷尽"检查。

---

## 5. 外科手术式修改 (Surgical Changes)

**只触碰必须触碰的部分。只清理你产生的垃圾。**

- **风格匹配：** 严格遵循既有代码风格，即使你个人并不认同。
- **严禁顺手牵羊：** 禁止"顺便"重构相邻无关代码、修改注释或重排版。
- **清理责任：**
  - 必须移除因**本次修改**导致的冗余导入（Imports）、变量或函数。
  - 除非被明确要求，否则不要清理代码中原有的"遗留垃圾"。
- **追溯标准：** Diff 中的每一行变更都必须能直接追溯到当前的 Task。Commit Message 中，强制携带 `spec.md` 中对应规则的章节号或 Task ID。

---

## 6. 技术硬约束 (Hard SOP)

- **表格驱动测试：** 凡涉及多分支、多输入逻辑，必须使用表格驱动风格。
- **Fail-Fast：** 禁止吞错、禁止返回语义不明的 `null`、禁止在错误状态下继续运行。错误处理必须遵循已评审文档中的契约：业务规则不满足时，应按 `spec.md` 返回明确且可观察的结果；外部依赖失败时，应按 `spec.md` / `plan.md` 执行显式的失败、重试、降级或中止策略；仅当系统规则被破坏、出现数据损坏风险或进入未定义状态时，才必须立即中止当前流程或使系统崩溃（Crash）。
- **同步律：** 提交代码前，必须自检并同步更新 `spec/plan/tasks`。

---

## 7. AI 提交自检清单 (Self-Checklist)

AI 在输出最终结果前，必须内部检索此清单：
- [ ] 我是否在没理解需求的情况下就跳到了编码阶段？
- [ ] 我是否引入了不必要的抽象或配置？
- [ ] 我的修改是否超出了 Task 的最小范围？
- [ ] 我是否为当前任务类型提供了匹配的验证证据？
- [ ] 若任务属于新增功能或 Bug 修复，我是否先写了失败测试？
- [ ] 我是否同步更新了相关的真理文档？

## 8. 一句话原则

**先以怀疑论者的身份审视需求，再以外科医生的手感精准编码，最后以法官的姿态验证交付。**
