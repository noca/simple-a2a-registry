# AI Agent 操作指令 — AI Coding 最佳实践 (SOP)

> 本文件详细定义了在不同场景（新项目初始化、既有项目加固规范化、Bug修复、架构重构等）下 AI Agent 必须履行的标准作业程序（SOP）。

---

## 1. 顶层场景判断逻辑

根据当前工作空间的状态，AI Agent 必须自主判定当前运行的分支流程，绝不允许走捷径：
* **分支 A — 新建项目初始化**：当前工作空间为空，或者用户发出 "创建新项目"、"从零构建" 的指令。
* **分支 B — 现有项目规范化**：当前工作空间已有大量业务代码，用户要求 "规范已有项目"、"资产接入与治理加固"。

> ⚠️ **最高前提**：在任何开发活动中，`spec.md` 都是业务真相的唯一容器，`plan.md` 都是技术实现蓝图的唯一容器，代码实现必须同时满足两者。

---

## 2. 分支 A：新建项目初始化（SDD 六步法）

该场景下严格遵循 **SDD (Spec-Driven Development, 规格驱动开发)**，按如下漏斗形时序链路推进：

```text
specification → planning → tasks → implementation → verification → delivery
       │              │         │               │               │            │
  1-需求澄清      2-技术方案  3-任务拆解      4-逐Task实现      5-自动化验证   6-交付报告
```

### A-1 需求澄清
* **目标**：将用户的模糊想法或简短描述，收敛为无歧义、高密度的 `spec.md`。
* **交付件**：`spec.md`（需包含背景、术语表、业务不变量、带 `SCN-xxx` 场景编号的核心逻辑流、Given/When/Then 验收标准、明确列出 Out-of-Scope 的业务边界）。
* **铁律**：此阶段严禁涉及任何具体的框架、数据库等技术方案设计。

### A-2 技术方案设计
* **目标**：基于已评审确认的 `spec.md`，输出可落地的技术实施蓝图 `plan.md`。
* **交付件**：`plan.md`（需包含对 spec 的完备性审查结论、核心领域模型建模、3-5 个带权衡理由的重大架构决策、前后端契约规范、非功能性高可用设计、测试验证策略与阶段路径）。
* **铁律**：架构设计不得臆造超出 `spec.md` 的业务功能，且每个决策需标记其出处（Spec 约束、宪法原则或假设）。

### A-3 任务解耦拆解
* **目标**：将 `plan.md` 转变为一系列原子、串行的 Task 清单 `tasks.md`。
* **交付件**：`tasks.md`（每个任务粒度在 S/M，强制禁止 L，需带明确的 Phase 与前置依赖，杜绝循环依赖）。
* **铁律**：任务编排顺序必须严格满足 **TDD 先行**（测试骨架 -> 生产代码 -> 集成验证）。

### A-4 逐任务实现
* **目标**：按 `tasks.md` 的 ID 顺序，每次仅领用并完成一个 Task。
* **铁律**：代码输出前，必须先声明假设和权衡。修改代码时必须在同一 turn 输送相匹配的失败测试用例。
* **熔断机制**：只要遇到 spec 缺失、文档与技术方案冲突、技术不可行或依赖未就绪等任意不确定性状态，**Agent 必须立即熔断停止**，将阻断点回推至文档澄清，严禁在代码层擅自脑补。

### A-5 交付与规范固化
* **目标**：项目完成实现后，固化持久开发约束。
* **交付件**：
  1. 生成完全定制的 `CLAUDE.md` 项目 AI 交互规范。
  2. 复制并合并 `governance/constitution.md` 与 `governance/AGENTS.md` 至项目。

### A-6 交付报告输出
* 按照标准化交付等式，向用户呈现应用资产树、用例场景条数、技术决策数、自检结果并说明验证证据。

---

## 3. 分支 B：现有项目规范化 (当前执行分支)

针对已有成熟代码库进行资产治理接入：

* **B-1：只读扫描**：使用 `prompts-extras/repo-analysis.md` 对项目全貌（架构、边界、包冲突、质量短板）进行全面评估，生成《项目整体认知报告》（写入 `docs/harness-repo-analysis.md`）。
* **B-2：生成 AI 开发规范**：根据项目具体技术栈适配通用模版，合并或全新生成 `CLAUDE.md` 写入项目根目录，并明确指出更新点。
* **B-3：治理文件引入**：将 `constitution.md` (开发宪法) 与 `AGENTS.md` (协作标准) 固化合并入项目治理目录（如 `docs/governance/`）。
* **B-4：输出治理报告**：列出风险点清单、已落地的规范资产、推荐后续改进项。

---

## 4. 特殊异常处理与硬性原则

### 4.1 Bug 修复 (使用 prompts-extras/debug.md)
* **原则**：严禁未经定位直接修改。
* **流程**：先进行 SDD 溯源和根因分析。若修改会影响原定契约，必须回文档层修复 Spec；若仅为实现缺陷，必须先编写能够 100% 重现该缺陷的**失败测试用例**，完成修复后确保测试与所有回归测试全绿通过。

### 4.2 变更影响分析 (使用 prompts-extras/change-impact.md)
* 在对复杂既有逻辑修改前，必须完成外科手术式的物理影响面分析。圈定涉及修改的类、函数深度，列出隐性级联级联风险，制定精密的回滚与测试靶向验证策略。

### 4.3 全局熔断条件 (熔断后必须停止并报错)
1. `spec.md` 内部存在逻辑冲突或业务不变量歧义。
2. `spec.md` 与技术方案 `plan.md` 存在冲突且无法通过架构规整消除。
3. `tasks.md` 出现不可解的前置依赖、循环锁。
4. 用户强制命令跳过测试、跳过方案设计或进行违反开发宪法的非安全修改。
