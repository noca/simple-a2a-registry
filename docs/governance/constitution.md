# 项目开发宪法 (Project Development Constitution)

> **版本：** 2.2
> **地位：** 本文件定义本项目的最高工程哲学与协作边界。它约束所有技术决策、代码评审与 AI 协作的工作方式，但不直接替代业务规格本身。
> **治理裁决顺序：** `constitution.md > AGENTS.md > spec.md > plan.md > tasks.md > 既有代码`。
> **事实来源顺序：** 业务行为以 `spec.md` 为唯一事实来源；技术实现以 `plan.md` 为唯一事实来源；代码必须同时服从已评审的 `spec.md` 与 `plan.md`。

---

## 1. 真理原则 (Truth)

* **1.1 SSOT (唯一事实来源)：** 经过评审的文档是真理的容器。其中，`spec.md` 定义业务规则与外部可观察行为，`plan.md` 定义实现方案与技术约束，代码仅是真理在特定技术栈上的投影。
* **1.2 同步律 (Sync Law)：** 文档与代码必须同频演进。任何导致文档过时的代码提交均视为违规，评审应直接予以拒绝。
* **1.3 诚实原则：** 严禁为了通过评审而粉饰文档。严禁"先写代码、后补文档"的造假行为；若实现改变了设计，必须先回溯修正文档。
* **1.4 边界原则：** `constitution.md` 与 `AGENTS.md` 负责规定过程、质量与协作纪律，不直接改写 `spec.md` 中已明确的业务规则。若业务规则与上位治理原则冲突，必须停止编码，先修正文档并重新评审，严禁在代码层静默裁决。

## 2. 测试先行铁律 (Test-First Imperative)

* **2.1 验证先行：** 所有新功能开发或缺陷修复，必须以编写一个（或多个）失败的测试用例为逻辑起点。凡改变外部可观察行为或关键契约的实现，均适用本条。
* **2.2 测试即契约：** 当任务涉及业务规则、接口行为、边界条件或回归风险时，测试用例是 `spec.md` 规则的物理体现。代码必须通过契约验证方能进入交付链路。
* **2.3 等价验证：** 对纯文档、纯配置、纯脚手架或不改变既有行为的结构性整理，可不强制先写失败测试，但必须提供与任务性质匹配的验证证据。

## 3. 演进原则 (Evolution)

* **3.1 先思后行：** 逻辑的纠错成本在 Spec 阶段最低，在 Code 阶段最高。任何模糊的需求必须在文档层击碎，严禁带入编码阶段。
* **3.2 闭环修正：** 当实现过程发现需要改变业务规则、外部可观察行为、关键接口契约或核心技术约束时，必须先修正 `spec.md` / `plan.md` 并重新评审。纯实现层问题应在不违背已评审文档的前提下直接解决，严禁将实现阻塞一律升级为文档变更。

## 4. 极简原则 (Parsimony)

* **4.1 YAGNI (你不需要它)：** 严禁为"假设的未来"增加复杂性。Spec 必须定义清晰的业务边界，代码必须在边界内保持克制。
* **4.2 简单胜于巧妙：** 拒绝任何形式的炫技。代码的可读性与可维护性永远高于执行效率，除非该点已被证明是系统的性能瓶颈。

## 5. 质量与安全 (Integrity)

* **5.1 交付等式：** **交付 = 与任务类型相匹配的变更 + 验证 + 权威文档同步**。新增功能与缺陷修复必须包含生产代码、测试代码与运行验证；纯文档、配置、CI 或脚手架任务可按任务性质提交对应工件，但不得省略验证与同步。
* **5.2 失败优先 (Fail-Fast)：** 异常必须被显式定义、显式暴露、显式处理，严禁沉默失败、状态污染或带病运行。

## 6. 责任共担 (Accountability)

* **6.1 人机协作协议：** AI Agent 负责逻辑生成与自动化验证，人类开发者负责最终决策、架构审美与边界把控。
* **6.2 零遗迹责任：** 不得以"历史原因"或"工具生成"为由规避合宪性责任。任何接触代码的人都有义务将其维护至合宪状态。
